## Issues found

### 1. 🐞 CRITICAL — Re-index loses resolved cross-file call edges when a producer is skipped
**File:** `crates/kernava-indexer/src/builder.rs` (the uncommitted shared-registry change)
**Symptom:** on the second `index_full_with_config` run, if a producer file (e.g. `lib.ts`) is
skipped (oversized / unreadable / parse-error), every consumer's cross-file call edge into
`lib.greet` silently regresses from *resolved* (target_id set) to *unresolved* (NULL target).
The graph rots without any signal to the user.
**Root cause:** the new optimisation replaces the per-file `build_registry(store, ...)` call
(which scanned the store for ALL existing symbols every file) with a single
`resolver::FunctionRegistry::new()` that starts the run **empty** and is only populated by
files indexed *this run*. `resolve_calls` returns `None` for any target whose symbol is not in
the in-RAM registry, and the store-fallback inside `index_file_inner` only fires *after*
resolution has already returned a `target_qualified` — so a skipped producer's symbols vanish
from resolution entirely, and the resulting edge is written with `target_id = NULL`.
This is a genuine regression vs the pre-change code (`index_file_with_config` still uses
`build_registry`, so the incremental/watcher path is unaffected — only `index_full`).
Repro test added: `builder::tests::test_index_full_skipped_producer_keeps_resolved_edges_on_reindex`.
**Fix applied (minimal, preserves the O(N) win):**
- `index_full_with_config` now seeds the registry ONCE from the store via `build_registry(store, "")`
  instead of `FunctionRegistry::new()`. One O(N) scan up-front, not N per-file scans — so the
  complexity class of the optimisation is intact, and skipped files keep their symbols
  resolvable by later consumers.
- `resolver::FunctionRegistry::register` is now **idempotent**: re-registering a qualified
  name that already exists overwrites the entry in place (same index) and does NOT append a
  duplicate to `by_simple`. Required so that re-registering the seeded entries with this run's
  fresh extractions doesn't make `try_global_unique` (which gates on `by_simple.len() == 1`)
  mistakenly see a duplicate and stop resolving. Also prevents unbounded `symbols` Vec growth
  across re-index loops.
  SAFETY note: when the `qualified_name` matches an existing entry the `name` field is also
  unchanged (the qualified name embeds the simple name as its last segment), so `by_simple`
  needs no update in the overwrite path.

### 2. ⚠️ MEDIUM — `topo_sort` cycle handling is O(N²) and order-dependent on cycle membership
**File:** `crates/kernava-indexer/src/builder.rs:380-389`
**Symptom:** files in an import cycle are detected via `result.contains(f)` which is a linear
scan inside a loop — O(N²) on cyclic graphs. On a large project with many cycles this is real
('slow' rather than 'broken'), and the appened order is the *sorted* file order, which is
arbitrary w.r.t. resolution quality.
**Fix (not applied — ponytail: defer until a project actually hits cycle-order resolution issues):**
colour/mark visited files in the leftover loop with a `HashSet` and append in dependency-aware
order. The cycle case is already noted as best-effort; no correctness regression today because
the seeded registry (fix #1) means cross-file resolution no longer depends on topo order being
perfect.

### 3. ⚠️ MEDIUM — `index_full_with_config` double-parses every file
**File:** `crates/kernava-indexer/src/builder.rs:285 + `build_import_deps``
**Symptom:** each file is parsed twice — once in `build_import_deps` (for the topo graph) and
once again inside `index_file_inner`. Marked with a `ponytail:` comment; harmless on small/medium
repos but 2× parse CPU on huge ones.
**Fix (deferred per existing ponytail note):** cache the parse trees / extractions from the first
pass and thread them into `index_file_inner`. Not urgent given tree-sitter speed.

### 4. ⚠️ LOW — `build_import_deps` skips the `max_file_size` gate
**File:** `crates/kernava-indexer/src/builder.rs:306-344` (already noted in code comment)
**Symptom:** an oversized file is read + parsed for import-graph extraction even though
`index_file_inner` will later bail on it. Correctness is fine (file still skipped at index time)
but it's a wasteful read for big binaries.
**Fix (deferred per existing ponytail note):** thread `config` into `build_import_deps` and
metadata-gate before reading.

### 5. ⚠️ LOW — Stack-overflow mitigated with a 256 MiB thread rather than iterative walk
**File:** `crates/kernava-server/src/lib.rs:190-217`
**Symptom:** `index_cmd` spawns a thread with `stack_size(256 MiB)` to survive deeply-nested
tree-sitter ASTs from C/C++ preprocessor-heavy headers, which overflow the 8 MiB default stack
during the recursive `extractor::walk()`. The comment correctly flags this as a ponytail that
should be fixed by converting `walk()` to an iterative work-stack loop.
**Risk:** the 256 MiB reservation is avertable OS-growth that may fail on memory-constrained
containers; the real bug is the unbounded recursion in `extractor`.
**Fix (deferred per existing ponytail comment):** iterative `VecDeque<Node>` traversal in
`extractor.rs`. Out of scope of this review's fix pass but should be the next correctness
investment if header-heavy C/C++ projects are supported.

### 6. ⚠️ LOW — `by_simple` can hold stale indices after fix #1's in-place overwrite
**File:** `crates/kernava-indexer/src/resolver.rs`
With the idempotent `register`, the `by_simple[name]` Vec index for an overwritten entry stays
correct (same index, same name). However `find_simple` still returns `&[usize]` whose underlying
`SymbolDef` was replaced — this is the desired behaviour (fresh definition) and matches how the
callers use it. No action needed; flagged for audit awareness only.

### 7. ⚠️ LOW — Watcher reading store while handler may be writing is not documented in tool handlers
Already addressed by the mutex serialisation note in `spawn_watcher`. Reviewing the other tool
handlers (`index_project` in `handler.rs`) for the same assumption is future audit work, not a
bug found here.

## Follow-up: addressing the remaining issues (2026-07-15)

After the critical fix in §1, the remaining items #2 through #7 were addressed on the same day:

### #2 — `topo_sort` O(N²) cycle loop 🐞 FIXED
Replaced the `result.contains(f)` linear-scan tail loop in `topo_sort` with a `visited: HashSet<&PathBuf>`
tracked alongside Kahn's queue. Cycle membership is now checked in O(1), making the tail loop O(N)
total instead of O(N²) on cyclic graphs. Function renamed-comment updated accordingly.

### #4 — `build_import_deps` missing size gate ⚠️ FIXED
`build_import_deps` now takes `&IndexerConfig` and metadata-gates oversized / unreadable files
before reading+parsing them — matching the index pass's behaviour. Removes a wasteful read+parse of
files that will be skipped at index time anyway. Both callers (`index_full_with_config`,
`index_incremental_with_config`) updated. The old `ponytail:` note about this gap was retired.

### #5 — Recursive `walk` / `count_complexity` / `scan_requires` overflow 🐞 FIXED
Real-world severity: the recursive extractor `walk()` (and the inner `count_node()` in
`count_complexity`, plus `scan_requires` in `languages/ts.rs`) each used one native stack frame per
AST level. Deeply-nested ASTs (C/C++ preprocessor-heavy headers, generated/bundled JS, even
deeply-nested TS) overflow the 8 MiB default thread stack at a few hundred levels — and crucially
this is the stack used by the **server handler** (`index_project` MCP tool runs on the tokio worker
thread) AND the **file watcher thread**, not just the CLI which had the 256 MiB mitigation. So the
production paths were exposed to a hard process abort on user-reachable inputs.
Reproduced deterministically: `extractor::tests::test_extract_deeply_nested_functions_no_stack_overflow`
imports a 1200-deep nested TS function string and runs `extract()` on a DEFAULT-stack (8 MiB) thread.
The recursive impl aborts the test runner (exit 134); the iterative impl passes.
Fix:
- `extractor::walk` → `walk_one<'t>` plus a `VecDeque<(Node<'t>, Option<String>)>` work stack. parent_symbol
  is now `Option<String>` (owned on the work stack, `clone`d per push). Every per-language match arm
  preserves its body verbatim except that `walk(&child, ..., parent, ...)` calls become
  `work.push_back((child, ...))`. `walk_method` and `extract_variable` share the same work stack.
- `count_complexity` split into pure `score()` + iterative subtree scan (was recursive at every symbol).
- `languages/ts::scan_requires` (CommonJS require scan) converted to iterative, preserving the
  require-declarator early-exit (handle() returns true ⟹ skip descending into the declarator).
All 188 workspace tests pass, clippy clean. The 256 MiB stack_size mitigation in `index_cmd` is now
redundant for the walk path but left in place as belt-and-suspenders for any future depth regression.

### #3 — `index_full` double-parses every file (ponytail) → DEFERRED, deliberately
Reviewed. Tree-sitter parses the supported languages at ~50-200 MB/s; on a 5000-file project the
second parse ≈ 0.5s. Caching extraction results across the topo loop would require plumbing a
`HashMap<PathBuf, ExtractionResult>` (or storing parsed `Tree`s) and feeding pre-extracted state
into `index_file_inner`, with non-trivial concurrency semantics alongside the seeded-registry logic
in §1. Judged YAGNI for v1 — the parse CPU is dominated by the registry + store writes anyway, and
the existing `ponytail:` comment already documents the tradeoff in-tree. No change made. Revisit if
profiled parse becomes a real bottleneck.

### #6 — `by_simple` stale-index audit ⚠️ AUDITED — no action
Re-verified after the §1 idempotent `register`: when a qualified name already exists, `register`
overwrites `symbols[existing_idx]` in place and does NOT append to `by_simple`. Therefore every
`by_simple[name]` index stays valid and points at the freshest `SymbolDef`. The only theoretical
divergence — a symbol's simple name changing while its qualified name stayed the same — cannot occur
because the qualified name embeds the simple name as its last `.`-separated segment. Confirmed
correct, no action required.

### #7 — mutex-serialisation assumption in handlers ⚠️ AUDITED — no action
Audited `handler.rs`: every MCP tool handler (`index_project`, `get_index_status`, `search_symbols`,
`get_symbol`, `get_file_outline`, `find_references`, `find_definition`, `search_code`, `get_callers`,
`get_callees`, `get_call_path`, `get_impact_radius_tool`, `detect_dead_code`, `query`) acquires
`self.state.store.lock()` before touching the store. The file watcher holds the same mutex only for
its short `filter_changes`+`process` critical section. Note that `search_code` scopes its lock to just
the `SELECT path FROM files` fetch and drops it for disk I/O, re-acquiring it only for Phase 2 — so
concurrent client search_code calls don't block each other on disk I/O. The single-writer invariant
the `spawn_watcher` doc claims is upheld consistently. No bug found.
